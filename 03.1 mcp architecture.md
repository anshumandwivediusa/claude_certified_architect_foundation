# MCP Architecture: Host, Client, and Server Relationship

## Claude Architect Exam Notes


# Module Overview

The **Model Context Protocol (MCP)** is an open protocol that standardizes how AI models communicate with external tools, data sources, and services.

MCP follows a **Client–Server Architecture**, where the AI application (Host) creates **MCP Clients** that communicate with **MCP Servers**.

One of the most frequently tested Claude Architect concepts is:

> **One MCP Client communicates with one MCP Server.**

This design provides **isolation, security, modularity, and independent lifecycle management**.


# MCP Architecture Overview

```text
                 User
                   │
                   ▼
          Claude Desktop / IDE /
        Enterprise AI Application
             (MCP Host)
                   │
     ┌─────────────┼─────────────┐
     ▼             ▼             ▼
 MCP Client A  MCP Client B  MCP Client C
     │             │             │
     ▼             ▼             ▼
Filesystem     GitHub        Database
 MCP Server    MCP Server    MCP Server
```


# Three Main Components

| Component  | Responsibility                           |
| ---------- | ---------------------------------------- |
| **Host**   | Runs Claude and manages MCP clients      |
| **Client** | Maintains a connection to one MCP server |
| **Server** | Exposes tools, prompts, and resources    |


# 1. MCP Host

## Definition

The **Host** is the application that embeds or runs Claude and manages all MCP interactions.

Examples:

* Claude Desktop
* Claude Code
* Enterprise AI Platform
* Custom AI Application
* IDE Plugin
* Internal Chat Assistant

The Host:

* Receives user requests
* Runs Claude
* Starts MCP clients
* Manages sessions
* Coordinates tool execution

Think of the Host as the **operating environment** for Claude.


# Host Responsibilities

* Start Claude
* Load MCP configuration
* Create MCP clients
* Manage user sessions
* Route tool requests
* Handle authentication
* Display responses


# Host Architecture

```text
            User

              │

              ▼

         Claude Host

              │

     Runs Claude Model

              │

Creates MCP Clients
```


# 2. MCP Client

## Definition

An **MCP Client** is a communication component created by the Host.

Its only responsibility is to communicate with **one specific MCP Server**.

It acts as a protocol adapter between Claude and that server.


# Important Rule

> **One MCP Client connects to exactly one MCP Server.**

Example

```text
Filesystem Client

↓

Filesystem Server
```

Another

```text
GitHub Client

↓

GitHub Server
```

Clients do **not** share server connections.


# Client Responsibilities

* Establish connection
* Exchange MCP messages
* Send tool requests
* Receive responses
* Handle transport (stdio, HTTP, WebSocket, etc.)
* Manage connection lifecycle


# 3. MCP Server

## Definition

An **MCP Server** exposes external capabilities to Claude.

Examples

* Filesystem
* GitHub
* PostgreSQL
* Slack
* Jira
* Google Drive
* Custom ERP
* SAP

The server advertises what it can do.


# MCP Server Provides

* Tools
* Resources
* Prompts
* Structured responses

The server never runs Claude itself.


# Relationship Between Components

```text
              Host
                │
        Creates Clients
                │
        ┌───────┼────────┐
        ▼       ▼        ▼
     Client   Client   Client
        │       │        │
        ▼       ▼        ▼
     Server   Server   Server
```

Notice

Each client owns one server connection.


# Why One Client per Server?

This is one of the most important Claude Architect concepts.

Instead of

```text
One Client

↓

Many Servers
```

Claude uses

```text
Client A → Server A

Client B → Server B

Client C → Server C
```


# Benefits

## 1. Isolation

Each server operates independently.

Example

GitHub crashes.

Filesystem still works.


## 2. Security

Different servers have different permissions.

Example

Filesystem

↓

Local disk access

GitHub

↓

Repository access

No permission mixing.


## 3. Independent Authentication

Example

```text
GitHub

OAuth
```

Database

```text
Username/Password
```

Filesystem

```text
No Authentication
```

Each client manages only its own server's authentication.


## 4. Easier Lifecycle Management

One server stops.

Only one client reconnects.

Entire system continues.


## 5. Better Scalability

Need another server?

Simply create

```text
New Client

↓

New Server
```

No impact on existing connections.

# Common Misconception

❌ **Incorrect**

```text
One Client

↓

Filesystem

GitHub

Slack

Database
```

This creates a single communication channel for all services, making authentication, lifecycle management, and error isolation difficult.


✅ **Correct**

```text
Filesystem Client

↓

Filesystem Server

------------------
GitHub Client

↓

GitHub Server

------------------
Slack Client

↓

Slack Server
```

Each connection is independent.


# Why Not One Client for Everything?

If one client handled every server:

* Authentication becomes complex.
* A single connection failure could affect all services.
* Security boundaries become weaker.
* Debugging is harder.
* Scaling individual services is difficult.

Dedicated clients keep the architecture modular and resilient.


# Best Practices

* One client per server.
* Keep servers focused on a single domain or capability.
* Let the host manage client lifecycle.
* Authenticate each client independently.
* Isolate failures to individual connections.
* Use standardized MCP messages for communication.
* Monitor client-server health independently.


# Interview Questions

### Q1. What are the three main components of MCP architecture?

**Answer:**
The three main components are:

* **Host** – Runs Claude and manages MCP clients.
* **Client** – Maintains a connection to a single MCP server.
* **Server** – Exposes tools, resources, and prompts to Claude.


### Q2. What is the role of the MCP Host?

**Answer:**
The Host runs the AI application, manages Claude, creates MCP clients, maintains user sessions, routes tool requests, and coordinates communication with MCP servers.


### Q3. Why does each MCP client connect to only one server?

**Answer:**
A one-client-per-server design provides isolation, independent authentication, simpler lifecycle management, better fault tolerance, and easier scalability. Problems with one server do not affect communication with others.


### Q4. Can a Host communicate with multiple MCP servers?

**Answer:**
Yes. The Host can create multiple MCP clients, with each client connecting to exactly one MCP server. This allows Claude to interact with many external systems simultaneously.


### Q5. Does the MCP server run Claude?

**Answer:**
No. The MCP server only exposes capabilities such as tools, resources, and prompts. Claude runs in the Host, which decides when and how to invoke those capabilities.

### Quick Memory Formula

> **Host = Orchestrates**
> **Client = Communicates**
> **Server = Provides capabilities**

### One-to-One Rule

```
1 Host
   │
   ├── 1 Client → 1 Server
   ├── 1 Client → 1 Server
   └── 1 Client → 1 Server
```

> **Not:** `1 Client → Many Servers`


# Claude Architect Key Takeaway

For the Claude Architect exam, remember this core relationship:

* The **Host** runs Claude and manages the overall application.
* The **Host creates one MCP Client for each MCP Server** it needs to communicate with.
* Each **Client maintains a dedicated connection to a single Server**, providing isolation, security, independent authentication, fault tolerance, and scalability.
* **Servers expose capabilities; Hosts orchestrate their use.** This separation of responsibilities is a fundamental design principle of the Model Context Protocol.
