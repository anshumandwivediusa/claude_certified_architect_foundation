# Error Propagation from Sub-Agent to Coordinator

# Module Overview

In a **Multi-Agent Architecture**, not every delegated task succeeds. A sub-agent may encounter:

* Tool failures
* API timeouts
* Missing data
* Permission errors
* Invalid inputs
* Rate limits
* Internal execution errors

A well-designed system **does not simply report "Task Failed."**

Instead, the sub-agent returns a **structured failure context**, allowing the **Coordinator Agent** to understand the cause, assess the impact, and decide the next course of action.

This is a key reliability pattern in enterprise AI systems.

# What is Error Propagation?

## Definition

**Error propagation** is the process by which a sub-agent reports execution failures back to the coordinator using structured information, enabling informed recovery decisions.

Instead of simply saying:

```text
Task Failed
```

the sub-agent explains:

* What failed
* Why it failed
* Which tool failed
* Whether retry is possible
* What partial work completed
* Suggested recovery options


# Why Error Propagation Matters

Imagine three agents are working:

```text
Research Agent

Coding Agent

Database Agent
```

If Database Agent returns

```text
Failed
```

the coordinator knows nothing.

But if it returns

```text
Database Timeout

Retry Recommended

No Data Retrieved

Tool: SQL MCP
```

the coordinator can recover intelligently.


# Generic Failure (Poor Design)

```text
Coordinator

↓

Database Agent

↓

Task Failed
```

Questions remain unanswered:

* Why?
* Which tool?
* Temporary?
* Retry?
* Permission issue?
* Bad query?
* Missing data?

Coordinator cannot make an informed decision.


# Structured Failure (Recommended)

```text
Coordinator

↓

Database Agent

↓

Status : Failed

Reason : SQL Timeout

Tool : SQL MCP

Retry : Yes

Attempts : 2

Partial Result : None

Recommendation :
Retry in 10 seconds
```

Now the coordinator has actionable information.

# Components of Structured Failure Context

A good error report should contain:

| Field              | Purpose                                  |
| ------------------ | ---------------------------------------- |
| Status             | Success / Failed / Partial               |
| Error Type         | Timeout, Permission, Validation, Network |
| Error Message      | Human-readable explanation               |
| Tool Used          | Filesystem MCP, SQL MCP, GitHub MCP      |
| Retryable          | Yes / No                                 |
| Retry Count        | Number of attempts                       |
| Partial Results    | Any successful output before failure     |
| Suggested Recovery | Retry, alternate tool, ask user          |

# Error Categories

### Transient Errors

Usually recoverable.

Examples

* Timeout
* Network issue
* Rate limit
* Temporary service outage

Coordinator typically retries.


### Permanent Errors

Retrying will not help.

Examples

* Permission denied
* Invalid credentials
* Unsupported operation
* Missing resource

Coordinator changes strategy or informs the user.


### Validation Errors

Example

```text
Invalid SQL

Invalid JSON

Incorrect Parameters
```

Coordinator may regenerate the request before retrying.


### Business Logic Errors

Example

```text
Invoice already processed

Account closed

Duplicate order
```

These require business-level decisions rather than technical retries.

# Silent Failure (Bad Practice)

```text
Coordinator

↓

Database Agent

↓

Nothing Returned
```

Coordinator cannot distinguish between:

* Empty results
* Timeout
* Crash
* Permission issue

Silent failures make autonomous systems unreliable.


# Structured Failure vs Silent Failure

| Structured Failure            | Silent / Generic Failure |
| ----------------------------- | ------------------------ |
| Explains cause                | No explanation           |
| Enables recovery              | No recovery guidance     |
| Includes retry recommendation | No retry information     |
| Supports auditing             | Difficult to audit       |
| Eases debugging               | Hard to diagnose         |
| Enterprise-ready              | Poor reliability         |


# Recovery Strategies

The coordinator may choose to:

### Retry

Useful for transient failures.


### Re-Delegate

Assign another sub-agent.


### Use Another MCP Server

Example

```text

↓

↓

```


### Ask User

Example

```text

```


### Return Partial Results

Example

```text

Complete

GCP

```

The coordinator communicates what is available instead of failing completely.


# Best Practices

* Always return structured failure metadata.
* Distinguish transient from permanent failures.
* Include retry recommendations.
* Preserve partial results whenever possible.
* Log every failure for observability.
* Avoid silent failures.
* Use standardized error formats across sub-agents.
* Let the coordinator decide recovery actions.


# Coordinator Responsibilities

The coordinator should:

1. Receive structured error information.
2. Classify the failure.
3. Determine if retry is appropriate.
4. Choose an alternative strategy when needed.
5. Merge partial successes.
6. Inform the user only when necessary.
7. Record the failure for monitoring and auditing.

# Interview Questions

### Q1. What is error propagation in a multi-agent system?

**Answer:**
Error propagation is the process of communicating execution failures from a sub-agent to the coordinator using structured error information, enabling intelligent recovery instead of simply reporting that a task failed.


### Q2. Why is structured failure context preferred over generic failure messages?

**Answer:**
Structured failure context provides actionable information such as the error type, failed tool, retryability, partial results, and recovery suggestions. This allows the coordinator to retry, switch strategies, or continue with partial results. Generic failures lack this information and hinder autonomous decision-making.


### Q3. What information should a structured failure report include?

**Answer:**
A structured failure report should include:

* Execution status
* Error type
* Error message
* Tool or MCP server involved
* Retry recommendation
* Retry count
* Partial results (if any)
* Suggested recovery action


### Q4. How does the coordinator respond to transient versus permanent failures?

**Answer:**
For transient failures (such as timeouts or rate limits), the coordinator typically retries or waits before retrying. For permanent failures (such as permission errors or invalid requests), it chooses an alternative tool, requests additional information, or informs the user.


### Q5. Why are silent failures dangerous in enterprise AI systems?

**Answer:**
Silent failures provide no diagnostic or recovery information, making it impossible for the coordinator to distinguish between conditions like "no data," "timeout," or "permission denied." This reduces reliability, complicates debugging, and weakens auditability.

### Claude Architect Key Takeaway

For the Claude Architect exam, remember that **sub-agents should never fail silently**. They should return **structured failure context** containing enough information for the coordinator to make intelligent recovery decisions. The coordinator—not the sub-agent—is responsible for choosing whether to retry, re-delegate, switch MCP tools, return partial results, or request user intervention. This separation of responsibilities is essential for building robust, observable, and enterprise-ready multi-agent systems.
